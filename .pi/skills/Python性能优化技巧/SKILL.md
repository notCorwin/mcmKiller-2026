# 1. 适用范围与技能目标

## 1.1 角色定位

本技能将模型设定为"数学建模 + 高性能 Python3 + M# 芯片加速"领域的专家。在数学建模解题（含各类竞赛场景）中，应基于 Python3 生成数值计算脚本，并优先利用 M# 芯片及向量化/并行化技术，使最终脚本的整体性能不低于、且尽可能优于 MATLAB 实现。

## 1.2 适用场景

涵盖但不限于以下典型建模任务类别：

- 回归分析与曲线拟合（线性/非线性）
- 最优化问题（线性规划、非线性规划、组合优化）
- 微分方程数值解（ODE / PDE）
- 计算机仿真与数值模拟
- 蒙特卡洛方法与随机模拟
- 图论与网络分析
- 时间序列分析与预测

## 1.3 触发条件

当用户在 Claude Code 中提出涉及以下任一关键词的需求时，应加载本技能并按其规范执行：

- "数学建模"与高性能代码需求同时出现
- "M#" 或同类加速硬件提及
- 要求替代 MATLAB 实现
- 显式提出 Python 数值计算加速

# 2. 工作流程规范

## 2.1 任务澄清与性能目标确认

在生成任何代码之前，应向用户确认以下信息：

1. **模型类型**：回归/优化/ODE/PDE/仿真/图算法等。
2. **数据规模**：样本数、特征维度、迭代次数的大致数量级。
3. **性能目标**：给出可量化的指标，例如"比现有 MATLAB 版本快 2 倍"或"在 N 秒内完成"。
4. **现有代码**：是否已有 MATLAB 参考代码、Python 原型代码或 M# SDK 文档的文件路径。

## 2.2 硬件与软件环境确认

收集并记录以下环境信息：

1. M# 芯片的接口方式：Python 包（如 `msharp`）、C 接口或命令行工具。
2. 可用计算资源：GPU/协处理器、SIMD 向量化指令（NEON/AVX）、多核并行。
3. 可用 Python 数值计算栈：NumPy、SciPy、Numba、CuPy、JAX、PyTorch 等。

## 2.3 算法与数据结构设计

在算法设计层面，遵循以下优先级：

1. **降低算法复杂度**：优先从 \(O(n^2)\) 改进至 \(O(n \log n)\) 或更优。
2. **向量化表示**：使用矩阵/张量运算替代逐元素 Python 循环。减少解释器级循环是提升性能的关键手段。
3. **数据结构选择**：根据访问模式选择连续内存布局（行主序/列主序）、适当的数据类型与容器。

## 2.4 M# 加速路径选择与集成

按以下优先级选择加速路径：

1. **M# 官方 Python 接口**：如 `msharp`、`msharp.linalg`、`msharp.sim` 等。
2. **JIT 编译**：使用 Numba 或 JAX 将热路径编译为高效核函数。
3. **并行计算**：多进程/多线程/异步（注意 CPython GIL 限制，优先使用多进程或已释放 GIL 的 C/NumPy 层并行）。

## 2.5 项目骨架生成

输出代码时，应按以下模块结构组织：

- `config`：超参数与路径配置
- `data`：数据加载与预处理
- `model`：核心数学模型与方程定义
- `solver`：数值算法实现及 M# 加速接口
- `main` 或 `run_experiment`：实验组织与结果记录

## 2.6 基准测试与性能分析

自动生成基准测试代码，实现以下对比：

- Python + M# 版本与纯 Python/NumPy 版本的性能比较。
- 若用户提供 MATLAB 版本，预留接口供用户在本地进行时间对比。

## 2.7 迭代优化

依据性能分析结果，遵循以下循环进行持续优化：

1. 性能剖析（Profiling），定位瓶颈。
2. 将瓶颈代码改写为向量化或并行核函数。
3. 重新测试，重复上述步骤直至达标。

# 3. MATLAB 迁移原则

当用户提供 MATLAB 思路或代码时，应遵循以下迁移原则：

## 3.1 转换思路而非逐行翻译

不应将 MATLAB 代码逐行翻译为 Python。应首先提取问题的数学本质——矩阵形式、方程结构、迭代策略——再以 Python + M# 重新组织实现。

## 3.2 优先采用批量/矩阵操作

将 MATLAB 中蕴含的向量化理念在 Python 中贯彻实施：

- 使用 NumPy 的广播机制、矩阵乘法和批量运算。
- 对高性能敏感模块，进一步迁移至 M# 提供的矩阵/张量接口。

## 3.3 利用 JIT 与编译加速

针对 MATLAB 内置高度优化库与 Python 解释器开销之间的差距：

- 将频繁调用的内层循环编写为纯 NumPy 数组运算，利用底层 C/Fortran 优化。
- 或使用 Numba/JAX 等工具的 JIT 编译机制加速。

## 3.4 算法优先于微优化

优化顺序应为：先降低算法复杂度，再考虑缓存局部性、数据布局、向量指令等底层细节。

# 4. M# 芯片使用策略

> **注意**：本技能不预设具体的 M# SDK API 名称。实际使用时应根据项目中已有文件（如 `references/msharp-api.md`、`msharp_*` 包）适配真实接口。草稿中可使用占位符，并提示用户替换为实际 API。

## 4.1 热路径调用原则

始终尝试在以下热路径中调用 M#：

- 大规模矩阵运算：线性代数、特征值分解、SVD、FFT 等。
- 大规模蒙特卡洛样本计算。
- 大量重复的迭代更新：梯度下降迭代步、模拟时间步等。

## 4.2 数据拷贝最小化

避免在 M# 核心循环中频繁于 Python 与设备间拷贝数据。建议采用以下模式：

1. 一次性将大块数据传入 M# 设备内存。
2. 在设备上完成尽可能多的计算。
3. 最后将结果一次性取回。

## 4.3 数据精度选择

根据建模需求合理设定数值精度：

- 在保证数值精度的前提下，优先使用 `float32` 以提升吞吐量。
- 对病态问题或累积误差敏感的环节，局部使用 `float64`。

## 4.4 可调参数暴露

为 M# 加速相关选项提供清晰的配置接口，至少包括：

- `use_msharp: bool`
- `dtype: "float32" | "float64"`
- `num_workers: int`
- `device: "auto" | "cpu" | "msharp" | "gpu"`

# 5. 各类建模任务的具体策略

生成代码前应判断任务类别，并按以下对应策略生成代码结构与加速方案。

## 5.1 回归与拟合（线性/非线性）

**算法建议：**

- 线性回归、岭回归：优先调用 M# 线性代数库（如 `msharp.linalg.solve` 或 LAPACK/BLAS 兼容接口）。
- 非线性拟合：采用基于梯度或数值微分的迭代算法，将梯度计算与批量预测部署至 M#。

**代码结构：**

- `build_design_matrix(X_raw) -> X`：仅在 CPU 上进行一次预处理。
- `msharp_fit(X, y, config) -> params`：在 M# 上求解或迭代。

## 5.2 优化问题（线性规划、非线性规划、组合优化）

**线性/凸优化：**

- 若 M# 提供专用求解器接口，优先调用。
- 否则在 Python 中组织问题结构，将大规模矩阵运算下沉至 M#。

**非凸/组合优化：**

- 采用批量评估策略，将多个候选解同时在 M# 上计算目标值/梯度。
- 对随机/启发式算法（遗传算法、模拟退火等），将"群体评估"编写为 M# 可加速的张量操作。

## 5.3 微分方程（ODE/PDE）与仿真

**ODE：**

- 优先使用矢量化形式，一次性计算多个时间步的增量。
- 对于刚性系统或大规模系统，将核心右端项（RHS）函数编写为可在 M# 上一次性处理的"状态批量"。

**PDE/格点仿真：**

- 将网格状态存储为 M# 支持的张量，利用卷积或稀疏矩阵乘法加速离散算子。
- 每个时间步的更新应尽量压缩为单个或少数几个 M# 调用。

## 5.4 蒙特卡洛与随机模拟

- 所有随机样本生成与样本级运算应尽可能在 M# 上批量完成：一次性生成大批量随机数，采用向量化 payoff 计算函数。
- Python 端仅保留种子管理与结果汇总/可视化（可适度使用 `matplotlib` 等工具）。

# 6. 性能测试与验证规范

每次生成完整脚本或模块时，应同时生成最小可运行的性能测试样例，以便用户快速验证。

## 6.1 基准测试结构

提供一个 `if __name__ == "__main__":` 区块，包含以下内容：

- 随机或合成数据构造，规模可配置。
- 调用核心求解函数，含 `use_msharp=True` 与 `use_msharp=False` 的对比。
- 使用 `time.perf_counter()` 记录并打印耗时。

## 6.2 可扩展评测接口

将问题规模（样本数、维度、迭代次数）暴露为命令行参数或配置变量，使用户可通过调整参数测试扩展性。

## 6.3 数值正确性检查

- **有解析解或参考结果的任务**：提供误差指标（相对误差、RMSE、最大绝对误差），确认优化不破坏数值正确性。
- **无解析解的问题**：提供收敛曲线、残差变化或守恒量（如能量/质量守恒）等过程指标。

# 7. 代码质量与项目结构规范

## 7.1 通用质量标准

- 使用 Python3 标准语法，遵循 PEP8 编码风格。
- 所有函数应包含类型注解（type hints）与简洁的 docstring。
- 模块化组织，避免将核心逻辑直接写入全局脚本。

## 7.2 文件结构建议

| 文件 | 职责 |
|---|---|
| `config.py` | 超参数与设备配置（M# / CPU / GPU） |
| `data.py` | 数据加载与预处理 |
| `model.py` | 数学模型与方程定义 |
| `solver.py` | 数值算法实现（含 M# 加速接口） |
| `benchmark.py` | 性能测试与对比 |
| `utils.py` | 通用工具函数（定时器、日志等） |

## 7.3 M# 相关封装约定

- 所有 M# 相关调用应封装至独立模块，如 `msharp_backend.py` 或 `backends/msharp.py`。
- 提供统一抽象接口：

  ```python
  Backend = Literal["cpu", "msharp"]

  def matmul(a, b, backend: Backend = "msharp"): ...
  ```

- 此设计使用户可通过修改配置在 CPU 与 M# 之间切换，便于调试与性能对比。

- 某段代码在 MATLAB 中的对应操作（如矩阵乘法、ODE 求解器等）。
- Python + M# 版本在性能上优于 MATLAB 的理论依据（如更低级硬件调用、更强并行度、消除解释器循环等）。
